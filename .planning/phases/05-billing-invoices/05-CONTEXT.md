# Phase 5: Billing & Invoices - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can view **subscription billing history** (past payments and charges for their tenant’s Stripe subscription) and **obtain invoices** (PDFs generated by our system). This phase is scoped to subscription billing only — not AR invoices, client portal invoices, or other financial modules.

</domain>

<decisions>
## Implementation Decisions

### Where billing history appears
- **Separate "Billing" tab in Settings.** Do not add billing history to the existing Subscription page. Add a new Settings tab (e.g. "Billing") that hosts the billing history and invoice list. Subscription tab stays focused on current plan, renewal, Change Plan, and Manage Billing.

### Source of billing data
- **Stripe only.** Edge Function(s) call Stripe API to list invoices and payment attempts; the app displays that data. No new database tables for caching or syncing invoice data. Stripe is the single source of truth.

### How users get invoices
- **We generate our own invoice PDF** and offer "Download invoice" in the app. Do not rely on Stripe’s hosted invoice URL only; the system must produce a downloadable PDF (e.g. server-side or Edge Function) so users can download invoices from within the product.

### What billing history includes
- **Invoices + payment attempts.** Show both:
  - Subscription invoices (each billing event: date, amount, status).
  - Payment attempts where relevant (e.g. failed then succeeded), so users can see failed charges and subsequent success.

### Empty state
- **Show the billing section with an empty state** when there is no history (e.g. trial-only or new tenant). Use a clear message like "No billing history yet" so the place is visible and consistent; do not hide the section.

### Claude's Discretion
- Layout and density of the billing history list (table vs cards, columns shown).
- PDF layout and branding for generated invoices (within existing design system).
- Exact Stripe API usage (e.g. `invoices.list` + `paymentIntents` or equivalent) and how to map to "payment attempts" in the UI.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **SubscriptionPage** (`src/components/Settings/SubscriptionPage.tsx`): Card-based layout, useSubscription, handleManageBilling (Stripe Portal). New Billing tab can follow same Settings tab pattern and Card usage.
- **useSubscription** (`src/hooks/useSubscription.ts`): Reads `subscriptions` (tenant_id, stripe_customer_id, stripe_subscription_id). New hook or Edge Function will need stripe_customer_id (or subscription id) to fetch Stripe invoices.
- **Settings** (`src/pages/Settings.tsx`): Tab structure for Settings; add a "Billing" tab alongside Subscription.
- **create-billing-portal-session**: Already verifies tenant and returns Stripe Portal URL; pattern for tenant-scoped Stripe Edge Functions.

### Established Patterns
- TanStack Query for server state; Supabase Edge Functions for Stripe (create-checkout-session, create-billing-portal-session). New: Edge Function(s) to list Stripe invoices (and optionally payment attempts) and to generate/serve invoice PDF.
- i18n: All UI strings in locale JSON (e.g. subscription.json); new namespace or keys for billing (billing.json) in en-US, pt-BR, es-ES, fr-FR.

### Integration Points
- Settings route and tab list (new "Billing" tab).
- Stripe API: invoices.list (by customer), invoice PDF or line items for our own PDF generation; payment attempts via invoice or payment intents as needed.

</code_context>
